\chapter{Modeling and Design of the Proposed Solution}

\section{Introduction}

Canny TMS is a custom-built internal web application conceived
and developed to meet the specific operational needs of Canny Vision. Designing and
implementing this system from end to end was my full responsibility --- from requirements
analysis through architecture, data modeling, security design, and the final deployed
product. The final goal was to provide Canny Vision's teams with a secure,
self-hosted platform for managing epics, tickets, and team collaboration, while strictly
respecting the company's technical and security constraints.

\section{Requirements Analysis}

\subsection{Functional Requirements}

Canny Vision already operates its own Identity and Access Management (IAM) system built
on OAuth~2.1 with PKCE. Any new application had to integrate with this existing IAM rather
than introduce an independent authentication layer. Canny TMS therefore delegates all
authentication and token validation to the IAM and derives all identity information from
the claims carried in the resulting JWT access tokens.

The company also enforces a strict policy of using no external JavaScript dependencies in
its frontend applications. Canny TMS's user interface was therefore developed in pure
vanilla JavaScript, leveraging the company's own in-house MVP rendering engine. This
approach keeps the entire client-side dependency surface fully auditable and eliminates
the supply-chain risk associated with third-party packages.

Finally, the application had to conform to a zero-trust architecture: all data must reside
on Canny Vision's own infrastructure, and every protected operation must be validated
internally by the application itself.
\subsection{Non-Functional Requirements}

Beyond functional correctness, the following non-functional properties were required:
\begin{itemize}
    \item \textbf{Usability:} the interface must be clean, intuitive, and accessible to
          non-technical team members such as commercial and client-role users.
    \item \textbf{Responsiveness:} the application must adapt gracefully to different
          screen sizes and devices.
    \item \textbf{Scalability:} the architecture must support multiple concurrent workspaces
          and teams without degradation.
    \item \textbf{Maintainability:} the codebase must be structured to allow future feature
          additions with minimal rework.
\end{itemize}

\subsection{Constraints and Assumptions}

\begin{itemize}
    \item Authentication and user identity are fully managed by the external IAM; Canny TMS
          stores only a local projection of each user (username, full name, e-mail),
          synchronized at login time.
    \item The frontend may not import any third-party library at runtime; all tooling,
          including CSS compilation, must produce standalone artifacts.
    \item The system is deployed exclusively on Canny Vision's own servers; no data may
          be transmitted to external services.
\end{itemize}

\section{Global Architecture}

\begin{figure}[H]
    \centering

    \caption{High-level architecture of Canny TMS}
    \label{fig:architecture-overview}

The system is organised into three tiers. The \textbf{frontend} is the SPA running in the
user's browser --- a vanilla JavaScript application delivered as static files. The
\textbf{middleware} tier is \textbf{WildFly}'s embedded web server, Undertow, which serves
those static files directly from the filesystem, terminates TLS, enforces CORS and
security headers, and routes requests to the appropriate deployed module. The
\textbf{backend} tier consists of two WAR deployments running inside the same WildFly
instance: the \textbf{IAM module}, which handles authentication and token validation, and
the \textbf{API module}, which implements the business logic --- both backed by a
\textbf{MySQL} database whose connection pool is also managed by WildFly.

\subsection{Entity-Relationship Diagram}

The data model of Canny TMS is structured around seven core entities:
\texttt{Workspace}, \texttt{Actor}, \texttt{Membership}, \texttt{Epic}, \texttt{Ticket},
\texttt{EpicComment}, and \texttt{TicketComment}.
Their relationships and attributes are illustrated in Figure~\ref{fig:erd}.

\begin{figure}[H]
    \centering
    \includesvg[width=\textwidth]{figures/erd}
    \caption{Entity-Relationship Diagram of Canny TMS}
    \label{fig:erd}
\end{figure}

A \textbf{Workspace} is the top-level organizational unit. Each workspace contains
multiple \textbf{Epics} (high-level goals or milestones) and \textbf{Tickets} (atomic
work items). Access to a workspace is controlled by a \textbf{Membership} record, which
associates an \textbf{Actor} --- an authenticated user whose identity is projected from
the IAM --- with a workspace-scoped role (\texttt{TECH\_MANAGER}, \texttt{BUSINESS\_MANAGER},
\texttt{DEVELOPER}, \texttt{COMMERCIAL}, or \texttt{CLIENT}). Tickets may optionally be
grouped under an Epic to express their relationship to a broader objective. Both Epics and
Tickets support threaded comments (\textbf{EpicComment} and \textbf{TicketComment}), with
a maximum nesting depth of one reply level enforced at the application layer.

\section{Design Decisions}

\subsection{Application Server: WildFly}

Most web projects split their infrastructure into separate pieces: a reverse proxy like
Nginx in front, an application server behind it, and some kind of connection pool manager
on top. WildFly collapses all of that into a single platform, and that is precisely why it
was chosen here.

Its embedded web server, Undertow, handles everything the frontend needs from a server:
static file serving from a directory on disk, the 404-to-\texttt{index.html} fallback that
makes client-side routing work, TLS termination via the Elytron security framework (TLS
1.3 only, with strong cipher suites), HTTP-to-HTTPS redirects, HSTS, CORS policy
enforcement, and cache-control headers. None of those required any application code ---
they are all configured at the server level through WildFly's CLI.

The two application modules --- the IAM and the API --- are deployed as standard WAR files
into the same WildFly instance. The MySQL connection pool is managed through WildFly's
datasource subsystem and exposed to the application via JNDI. Access logging is
configured per virtual host.

Running everything under one platform means one log file, one configuration system, and
one deployment target. For a self-hosted, single-server setup, this is straightforwardly
simpler than maintaining separate components.




\subsection{CSS Framework: From Bootstrap to Tailwind CSS}

The initial frontend prototype was built with Bootstrap, which is Canny Vision's default
choice for rapid UI prototyping --- imported from a CDN and therefore free of server-side
build concerns. While Bootstrap proved adequate for early scaffolding, its opinionated
component model tends to produce a generic visual result, and its comprehensive stylesheet
ships a significant volume of unused CSS that negatively affects loading performance.

After analysis and with the approval of Mr.~Kaaniche, the project migrated to
\textbf{Tailwind CSS}. Tailwind's utility-first approach provides precise control over
the visual design without shipping unused style rules. Because no npm-based build pipeline
was permissible under the company's dependency policy, Tailwind was integrated via its
\textbf{standalone CLI}: a self-contained executable that compiles only the utility classes
actually referenced in the source files into a single artifact, with no Node.js or package
manager required at runtime~\cite{tailwindStandaloneCLI}.

\subsection{Authorization: Attribute-Based Access Control}

Standard Role-Based Access Control (RBAC) was insufficient for Canny TMS's authorization
requirements. The same authenticated user may hold different roles across different
workspaces, and resource ownership confers additional privileges beyond what a workspace
role alone would permit. A workspace member who created an epic, for example, retains
the right to modify it even when their workspace role would not otherwise allow such an
operation.

The authorization model therefore follows the \textbf{NIST SP~800-162} guide to
Attribute-Based Access Control~\cite{nistABAC}, which defines access decisions as a
function of \emph{subject attributes} (identity, workspace role), \emph{object attributes}
(resource type, ownership), and \emph{environment attributes}, evaluated against a
declarative policy. The implementation mirrors the XACML reference architecture:

\begin{itemize}
    \item a \textbf{Policy Enforcement Point} (PEP) intercepts each protected API call,
    \item a \textbf{Policy Decision Point} (PDP) evaluates the request against a
          workspace-scoped permission matrix,
    \item a \textbf{Policy Information Point} (PIP) resolves the runtime attributes
          needed for the decision (e.g.\ the caller's membership role, resource
          ownership).
\end{itemize}

The design was informed by a detailed study of \textbf{AuthZforce}, an open-source XACML
engine, whose source code and architecture served as a reference during the design phase.

The \textbf{Policy Administration Point} (PAP) --- which would allow non-technical
administrators to define and modify policies at runtime --- was intentionally left out of
scope. Given the fixed, well-defined permission model of Canny TMS, a dynamic PAP would
add significant complexity without a practical benefit.

\subsection{Progressive Web App Strategy}

Canny TMS is delivered as a Progressive Web App. The reasoning is simple: the people
using this tool are not always at a desk with a stable connection. Being able to browse
tickets on a phone with patchy Wi-Fi, or get a notification when something is assigned,
makes a real difference in day-to-day use.

The caching strategy is \textit{Network with Cache Fallback}. The Service Worker tries
the network on every request; if it fails, it returns the cached version. This is the
right fit for a read-heavy tool where most interactions are browsing boards and reviewing
ticket details.

Offline writes are handled by the \textbf{Background Sync API}. When a team member
updates something while offline, the change is saved locally in IndexedDB. The moment
connectivity is restored, the Service Worker sends the queued request to the server ---
no manual retry needed on the user's side.

For push notifications, the \textbf{Web Push API} is used. The server sends a
notification whenever a ticket is assigned to a user. This maps directly to the
\textit{Manage Tickets} and \textit{Manage Epics} use cases: the assigned team member
knows immediately, without having to keep the board open.

All of this runs on standard browser APIs, no extra dependencies added.

\section{Chapter Conclusion}

The requirements, architecture, and design choices covered in this chapter all trace back
to the same starting point: the constraints Canny Vision operates under. The dependency
policy drove the frontend approach, the existing IAM determined how authentication works,
and the multi-role access model made a simple RBAC scheme insufficient. Each decision ---
from the Tailwind standalone CLI, to WildFly as the single deployment platform, to the
ABAC policy engine, to the PWA caching and notification strategy --- has a concrete reason
behind it rather than being an arbitrary technical preference.
The next chapter moves to the implementation itself: the technologies selected, how the
components were built, and what the finished system looks like in practice.
